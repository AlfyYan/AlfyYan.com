Unity集成到iOS本地工程中

By AlfyYan

发表于 2016-09-07

软件环境

　　Unity3D: 5.4.0f3
　　Xcode: 7.3

准备工作

　　为便于管理，可先在桌面上新建文件夹Unity->iOS，再在该文件夹中新建两个子文件夹iOS和Unity,分别用于存放iOS原生工程和Unity导出的iOS工程。

1.iOS原生工程，简称iOS工程

　　用Xcode新建一个Single View Application工程, 名称随意，本文按照视频中命名为Unity5Native，存放在事先建好的iOS文件夹中。

2. Unity导出的iOS工程，简称Unity工程

用Unity导出iOS工程，存放在Unity文件夹中。

集成步骤

1. iOS工程中新建Vendor group

2. 添加Unity的文件至iOS工程中

　　将Unity工程下的3个文件夹Classes、Libraries、Data添加至iOS工程中，添加时注意选择Copy items if needed选项，Classes和Libraries文件夹选择Create groups, 而Data文件夹选择Create folder references选项。

　　添加完毕以后可以把Unity文件夹中的一些无关文件清理掉，它们会影响编译速度,删除时选择Remove Reference：

  • Classes->Native文件夹下的*.h文件
  • Libraries文件夹下的libil2cpp文件夹

    3. 添加相关框架

    　　在iOS工程中新建Frameworks group, 将需要添加的framework放在这个group下，至于需要添加哪些framework可以参见Unity工程的TARGETS -> General -> Linked Frameworks and Libraries，如图所示：

4. 配置Build Settings相关选项

（1） Enable Bitcode -> NO
（2） Other Linker Flags
添加-weak_framework CoreMotion -weak-lSystem

（3） Header Search Paths
添加以下选项：

  • "$(SRCROOT)"
  • "$(SRCROOT)/Unity5Native/Classes"
  • $(SRCROOT)/Unity5Native/Classes/Native
  • $(SRCROOT)/Unity5Native/Libraries/libil2cpp/include
  • $(SRCROOT)/Unity5Native/Libraries

注意：根据工程实际路径来配置，$(SRCROOT)是指.xcproj文件所在的路径，Classes和Libraries文件夹是在与Unity5Native.xcproj文件同级的Unity5Native文件夹下面的，所以SRCROOT下面要再添加Unity5Native层级。如果此处路径配置不正确，可能会出现诸如il2cpp-config.h file not found的编译错误。

（4）Library Search Paths
添加以下选项：

  • "$(SRCROOT)"
  • "$(SRCROOT)/Unity5Native/Libraries"

（5）Custom Compiler Flags -> Other C Flags
添加-DINIT_SCRIPTING_BACKEND=1
（6） C Language Dialect选择C99 [-std=c99]
（7） 新建pch文件：PrefixHeader.pch
（8） 设置Prefix Header的路径
根据pct文件所在路径实际设置，由于我是放在Supporting Filegroup组下面的，实际在与Unity5Native.xcproj文件同目录下的Unity5Native文件夹下。
（9）Precompile Prefix Header选择YES
（10） C++ Language Dialect选择C++11 [-std=c++11]
（11）Enable C++ Runtime Types选择NO
（12） Warnings-Objective C->Overriding Deprecated Objective-C Methods选择YES
（13） Warnings-Objective C->Unintentional Root Classes选择YES
（14） User-Defined
添加如下选项：

  • GCC_THUMB_SUPPORT->NO
  • GCC_USE_INDIRECT_FUNCTION_CALLS->NO
  • UNITY_RUNTIME_VERSION->5.4.0f3
  • UNITY_SCRIPTING_BACKEND->il2cpp

5. pch文件内容替换

将Classes文件夹下的Prefix.pch文件中的内容全部复制粘贴到PrefixHeader.pch文件的

#define PrefixHeader_pch
#endif /* PrefixHeader_pch */

之间，并且引入UnityAppController.h文件如图所示：

6. Supporting Files下面的main.m

  • 重命名成main.mm

  • 将Classes文件夹下的main.mm文件中的全部内容复制到Supporting Files文件夹下的main.mm文件中，并进行修改：

    //const char* AppControllerClassName = @”UnityAppController”;
    const char* AppControllerClassName = “AppDelegate”;

  • Build Phases中移除Classes下的main.mm

7. UnityAppController.h

作如下修改：
//inline UnityAppController GetAppController()
//{
// return (UnityAppController)[UIApplication sharedApplication].delegate;
//}

#import "AppDelegate.h"

inline UnityAppController*  GetAppController()
{
    AppDelegate *delegate = (AppDelegate *)[UIApplication sharedApplication].delegate;

    return delegate.unityController;
}

8. AppDelegate.h和AppDelegate.m

AppDelegate.h文件作如下修改：

#import <UIKit/UIKit.h>

@interface AppDelegate : UIResponder <UIApplicationDelegate>

@property (strong, nonatomic) UIWindow *window;
@property (strong, nonatomic) UIWindow *unityWindow;

@property (nonatomic, strong) UnityAppController *unityController;

- (void)showUnityWindow;
- (void)hideUnityWindow;

@end

AppDelegate.m文件作如下修改：

#import "AppDelegate.h"

@interface AppDelegate ()

@end

@implementation AppDelegate

- (UIWindow *)unityWindow {
    return UnityGetMainWindow();
}

- (void)showUnityWindow {
    [self.unityWindow makeKeyAndVisible];
}

- (void)hideUnityWindow {
    [self.window makeKeyAndVisible];
}

- (BOOL)application:(UIApplication *)application didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {
    // Override point for customization after application launch.

    self.window.backgroundColor = [UIColor redColor];

    self.unityController = [[UnityAppController alloc] init];

    [self.unityController application:application didFinishLaunchingWithOptions:launchOptions];

    return YES;
}

- (void)applicationWillResignActive:(UIApplication *)application {
    // Sent when the application is about to move from active to inactive state. This can occur for certain types of temporary interruptions (such as an incoming phone call or SMS message) or when the user quits the application and it begins the transition to the background state.
    // Use this method to pause ongoing tasks, disable timers, and throttle down OpenGL ES frame rates. Games should use this method to pause the game.

    [self.unityController applicationWillResignActive:application];
}

- (void)applicationDidEnterBackground:(UIApplication *)application {
    // Use this method to release shared resources, save user data, invalidate timers, and store enough application state information to restore your application to its current state in case it is terminated later.
    // If your application supports background execution, this method is called instead of applicationWillTerminate: when the user quits.

    [self.unityController applicationDidEnterBackground:application];
}

- (void)applicationWillEnterForeground:(UIApplication *)application {
    // Called as part of the transition from the background to the inactive state; here you can undo many of the changes made on entering the background.

    [self.unityController applicationWillEnterForeground:application];
}

- (void)applicationDidBecomeActive:(UIApplication *)application {
    // Restart any tasks that were paused (or not yet started) while the application was inactive. If the application was previously in the background, optionally refresh the user interface.

    [self.unityController applicationDidBecomeActive:application];
}

- (void)applicationWillTerminate:(UIApplication *)application {
    // Called when the application is about to terminate. Save data if appropriate. See also applicationDidEnterBackground:.

    [self.unityController applicationWillTerminate:application];
}

@end

9. ViewController.m

作如下修改：

#import "ViewController.h"
#import "AppDelegate.h"

@interface ViewController ()

@property (nonatomic, strong) UIButton *showUnityButton;

@end

@implementation ViewController

- (void)viewDidLoad {
    [super viewDidLoad];
    // Do any additional setup after loading the view, typically from a nib.

    self.view.backgroundColor = [UIColor blueColor];

    self.showUnityButton = [UIButton buttonWithType:UIButtonTypeSystem];
    [self.showUnityButton setTitle:@"Show Unity" forState:UIControlStateNormal];
    self.showUnityButton.frame = CGRectMake(0, 0, 100, 44);
    self.showUnityButton.center = self.view.center;
    [self.showUnityButton addTarget:self action:@selector(showUnityView:) forControlEvents:UIControlEventTouchUpInside];
    [self.view addSubview:self.showUnityButton];
}

- (void)showUnityView:(id)sender {
    [(AppDelegate *)[UIApplication sharedApplication].delegate showUnityWindow];
}

@end

iOS开发——Xcode

Xcode插件

  1. Alcatraz
    Alcatraz是Xcode中开源的包管理工具插件，可以让我们更轻松地管理各种插件，比如查找、安装、删除别的插件、模板、颜色主体，而不需要手动将插件文件复制到Xcode的插件目录下，Alcatraz帮我们做这项工作。

  2. FuzzyAutoComplete
    Xcode本身自带智能提示补全代码功能，但是前提是你得记得正确的方法头，比如-(void)tableView:(UITableView *)tableView cell才能匹配出方法，但是使用该插件，你只需要输入诸如-cellFor就可以匹配出来，模糊匹配非常强大。

  3. KSImageNamed
    KSImageNamed插件会自动弹出图片名称列表以供选择，并且显示缩略图，让我们能够方便地查看资源文件夹以查找合适的图片名称。
    顺便说下，这个插件貌似用Alcatraz无法安装。如果你也同样碰到Alcatraz安装此插件失败的问题，那么可以按照它的github页说明的那样，将项目下载下来build，再重启Xcode就可以了。

  4. VVDocumenter-Xcode
    VVDocumenter-Xcode是猫神出品，可以让你方便地添加Java doc风格的注释。

  5. SCXodeSwitchExpander
    虽然Xcode自带的智能提示功能清楚地列出了Switch语句每种情况下应该处理的方式，但是一个个填入枚举值仍然很繁琐。SCXodeSwitchExpander插件正是为此而生，它可以自动获得所有的枚举值并填入相应的位置，而我们只需要处理每个枚举值下面对应的逻辑即可。

  6. RTImageAssets
    该插件用来生成 @3x 的图片资源对应的 @2x 和 @1x 版本，只要拖拽高清图到 @3x 的位置上，然后按 Ctrl+Shift+A 即可自动生成两张低清的补全空位。当然你也可以从 @2x 的图生成 @3x 版本，如果你对图片质量要求不高的话。

The purpose of using IBOutlet and IBAction

IBOutlet和IBAction本质是Apple定义的宏，用来指代Interface Builder文件构建的界面上的变量和方法。比如界面上有一个UILabel控件，但是在xib对应的类文件中，你想动态改变这个label的文本，那就需要在类文件中引用界面上的这个label，IBOutlet的作用就是将两者连接起来，让其建立联系。

How to disable ARC for a single file in a project

【问题】项目开发中通常会用到第三方提供的源代码，但问题是一些比较老的开源项目可能使用的还是非ARC模式，那么我们怎么让ARC和非ARC的源文件同时存在于一个工程中呢？

【解决】Xcode项目中可以使用ARC和非ARC的混合模式。
　如果项目中使用的是非ARC模式，则为ARC模式的源文件加入fno-objc-arc标签；如果项目中使用的是ARC模式，则为非ARC模式的源文件加入fobjc-arc标签。
添加标签的方法：

  1. 打开：Target->Build Phases->Compile Sources
  2. 双击对应的.m文件
  3. 在弹出窗口中输入上面提到的标签fobjc-arc/fno-objc-arc
  4. 点击Done保存

Hexo搭建Github博客踩到的那些坑

By AlfyYan

发表于 2015-08-10

前言

　Hexo是由Node.js驱动的一款快速、简单且功能强大的博客框架，尽管网上已经有很多文章介绍如何利用Hexo搭建博客，但是我们在实践的过程中仍然会遇到很多问题。谨以本篇文章记录我在使用Hexo搭建博客、写博客遇到的问题。

1号坑-Hexo安装问题

问题描述

　首先，需要明确，对于mac用户，安装Hexo之前需要install：

  • Command Line Tools

Xcode安装之后，Xcode->Preferences->Download->Command Line Tools->Install，即可安装。

  • Node.js

Node.js官网是用pkg安装包安装的，然而treehouse的这篇tutorial推荐使用Homebrew命令行安装，理由如下：

（1） 用Homebrew安装不需要给予sudo权限，这样更安全；

（2） 用安装包安装的话还要将node的可执行路径添加到系统路径(system $path下)，对于熟悉Linux或者Unix的用户，也许很容易，然而对那些对命令行不熟悉的新手来说，这是一个很痛苦的过程，而Homebrew就不需要你去配置系统路径这些。。。

（3）Homebrew是mac下的开发人员必备安装工具啊！使用安装诸如Git、Ruby等工具都将大大简化，同时卸载这些工具也很方便。

一次遇到的三个问题

Question One

diff: /../Podfile.lock: No such file or directory
diff: /Manifest.lock: No such file or directory
error: The sandbox is not in sync with the Podfile.lock. Run 'pod install' or update your CocoaPods installation.

Question Two

ld: library not found for -IQKeyboardManager

Question Three

Pods was rejected as an implicit dependency for 'libPods.a' because its architectures 'XXX' didn't contain all required architectures

　第二天要面试，想给面试官展示曾经做过的项目，于是把项目打开运行，编译却没通过，遇到第1个问题，但还好错误说明中给出了如何解决的方法——Run ‘pod install’ or update your CocoaPods installation. 关闭项目执行”pod install”。PS：此处说个小技巧，通常我们在运行”pod install/update”命令的时候会很慢,原因在于执行这两个命令的时候会升级CocoaPods的spec仓库，加个参数就会少执行这一步，速度也会相应提升：
pod install –verbose –no-repo-update
pod update –verbose –no-repo-update

　上述命令执行完之后，再次编译运行，哇咔咔。。。仍然是buil d failed o (╯□╰)o，但是这次error原因变了，变成第2、3个 错误了，谷歌ld: library not found for - IQKeyboardManager,网上有很多关于ld: library not found for...错误的解决方案，比如此处，但是我的所有第三方库都是用CocoaPods管理的，应当与链接引用没关系，搜寻各处无果。。。重新再返回xcode查看错误时，发现还有一个隐晦的warning，即上述第3个问题，(⊙o⊙)…貌似libPods.a库有问题啊，再去找谷歌，并间接寻到了万能的stackoverflow上——就是这儿，按照上面所说去修改项目，build——>run——>泪奔~~o(>_\<)o ~~讨厌的红色叉叉终于没了。。。
　 　

15-11-3遇到的问题

CocoaPods国内淘宝镜像失效

　今天在运行pod install命令的时候，突然出现command not found，自己非常确定之前已经安装过，但是既然它提示了，那么就重装一遍吧。但是执行sudo gem install cocoapods时又报错了：

ERROR:  Could not find a valid gem 'cocoapods' (>= 0), here is why:
   Unable to download data from http://ruby.taobao.org - bad response Not Found 404 (http://ruby.taobao.org/latest_specs.4.8.gz)

“not found 404”很明显是淘宝镜像失效了，而且现在使用的VPN命令行里也可以翻墙，所以决定删除原淘宝镜像(http://ruby.taobao.org)， 重新添加原来镜像地址(https://rubygems.org)。 按照当初装CocoaPods的教程, 可以知道删除镜像和添加镜像的方法分别如下:

gem sources —-remove http://ruby.taobao.org/
gem sources -a https://rubygems.org

但是执行gem remove命令时，却出现如下错误：

source http://ruby.taobao.org/ not present in cache

一直删除不掉这个淘宝镜像，最终搜索了下，在SegmentFault上找到答案，竟然是一个”/”的问题：

gem sources —-remove http://ruby.taobao.org

😥 😥 😥 😥!!
　另外镜像失效的原因是淘宝把原先的http协议镜像改为https协议镜像，所以只要将镜像地址更新为”https://ruby.taobao.org”就好了。
